# Implementation Plan

- [ ] 1. 基础：用户反馈消息载体
- [x] 1.1 在基础模板中渲染闪现消息
  - 在 base.html 增加 Django messages 渲染区块，使后续「替换 / 新建 / 失败」消息可在跳转后的页面上显示。
  - 消息按等级（success / error）区分展示样式，与现有错误样式风格一致。
  - 观察完成：携带闪现消息跳转到任意页面（如文档详情页）时，页面顶部能看到对应消息文本。
  - _Requirements: 3.3, 4.1, 4.2_
  - _Boundary: base.html template_

- [ ] 2. 核心：同名替换流程
- [x] 2.1 抽取可复用的 ingestion 步骤
  - 将现有上传中的「文本抽取 → 分块 → 嵌入生成」整理为一个可复用的内部步骤，失败时抛出异常，成功时返回分块与对应向量。
  - 新建路径改用该步骤，保持现有新建行为与结果不变（回归无差异）。
  - 观察完成：上传新文件仍正常创建文档并生成分块，现有上传相关测试全部通过。
  - _Requirements: 2.2, 3.1_
  - _Boundary: UploadView_

- [x] 2.2 实现同名检测与分支
  - 在上传处理中按文件名查找现有文档；存在则进入替换路径，不存在则进入新建路径。
  - 同名判定仅依据文件名，不考虑文档种别。
  - 观察完成：上传与现有文档同名的文件不再新增文档行数；上传不同名文件仍新增一行；同名但不同种别仍判定为同名。
  - _Requirements: 1.1, 1.2, 1.3_
  - _Boundary: UploadView_
  - _Depends: 2.1_

- [x] 2.3 实现替换持久化（复用记录、重建分块）
  - 在单一事务内：删除目标文档的旧分块、基于新内容写入新分块、保存新原本 PDF 并切换文件引用、更新种别/分块数/状态/错误信息，并删除其余同名重复文档（连带其分块）。
  - 旧物理文件在事务成功提交后删除（事务提交回调），不在事务内删除，以免回滚时误删既有文件。
  - 保持目标文档标识（document id）不变。
  - 观察完成：替换后目标文档 id 不变、旧分块全部消失、新分块数等于新内容分块数、该文件名对应的有效文档仅剩一条、详情页展示替换后的最新文件与种别。
  - _Requirements: 1.4, 2.1, 2.2, 2.3, 2.4, 2.5_
  - _Boundary: UploadView_
  - _Depends: 2.2_

- [x] 2.4 保证替换失败时的数据安全
  - 将易失败的 ingestion 步骤置于任何破坏性写入之前；破坏性写入全部包含在事务中，失败即整体回滚。
  - 替换失败时不修改既有文档与分块、不写入既有文档的错误信息，改用闪现消息提示失败原因，并跳转到该既有文档的详情页。
  - 观察完成：替换一个无可抽取文本（图像型）或在嵌入阶段失败的文件后，既有文档与其分块保持替换前状态，页面显示失败原因消息。
  - _Requirements: 3.1, 3.2, 3.3_
  - _Boundary: UploadView_
  - _Depends: 2.3, 1.1_

- [x] 2.5 实现成功反馈与跳转
  - 替换成功时提示「替换现有文档」，新建成功时提示「新建文档」，两种情况均跳转到对应文档详情页。
  - 观察完成：替换成功后详情页显示「替换」消息，新建成功后显示「新建」消息，且都停在对应文档详情页。
  - _Requirements: 4.1, 4.2, 4.3_
  - _Boundary: UploadView_
  - _Depends: 2.3, 1.1_

- [ ] 3. 验证：上传替换测试
- [x] 3.1 同名检测与分支测试
  - 覆盖：同名→不新增行、不同名→新增行、同名不同种别→仍替换。
  - 观察完成：对应测试在测试套件中通过。
  - _Requirements: 1.1, 1.2, 1.3_

## Implementation Notes
- 実装は views.py の `UploadView.post` に集約。`_ingest`（抽出→分割→埋め込み、失敗時に例外）と `_replace_document`（単一トランザクションで旧 chunk 削除→新 chunk 生成→新ファイル保存→フィールド更新→同名重複文書削除、旧物理ファイルは `transaction.on_commit` でコミット後削除）を追加。
- タスク 3.1〜3.4 はテストカバレッジ確認タスク。2.1〜2.5 が TDD-first で実装したため大半は既存テストで充足済み。重複した subagent 起動を避けるため 1 つのカバレッジ補完タスクに統合し、要件 2.5（検索整合: 旧 chunk 削除を永続層で検証）と 2.4（詳細ページ最新化）の明示テストを追加した。
- 要件 2.4（解決済み）: 当初 `document_detail.html` は category を表示していなかったため、種別の可視表示を追加（「種別」行、未設定時は「未分類」）。テストは描画本文に新種別の表示名が現れ旧種別が消えることまで検証する。これで詳細ページが置き換え後の最新の種別も可視で反映する。
