# 需求文档

## 简介

本功能为既有 PDF 知识库（Django + pgvector）构建一个 TypeScript（Bun + Hono）的 AI Agent 服务。企业内部员工（HR、法务、业务等非工程师）在网页聊天界面用自然语言提问，Agent 通过 LLM 的 Function Calling 自主判断并调用知识库工具，循环执行后彙整出带来源（文件名 + 片段 + 原文链接）的自然语言回答，免去手动关键字搜索。同一套工具逻辑同时以 MCP Server 形式暴露，供其他 MCP 客户端复用。

检索能力（嵌入、向量搜索、文档存储）由既有 `pdf-knowledge-base` 规格通过只读 JSON REST API 提供，本规格不重复实现，仅作为该 API 的消费方与编排层。

## 边界上下文

- **范围内**：自然语言聊天问答、会话内多轮对话（内存内）、Agent 工具调用循环（LLM Function Calling）、三个知识库工具（语义搜索 / 文档列表 / 文档详情）、带来源与原文链接的回答、思考过程日志、Web 聊天界面、MCP Server
- **范围外**：嵌入与向量搜索逻辑（保留在既有 Python 侧）、知识库 REST API 的实现（由 `pdf-knowledge-base` 提供）、PDF 上传与 ingestion、用户认证与权限、对话历史的持久化存储（仅内存内、重启即失）、原文页级深链
- **相邻期望**：
  - 本服务所有知识库访问均经由既有知识库 REST API（`POST /api/search/`、`GET /api/documents/`、`GET /api/documents/<id>/`）；其请求/响应契约（含原文 `file_url` 字段）由 `pdf-knowledge-base` 定义，本服务严格对齐
  - LLM 须支持 Function Calling（工具调用）协议；具体供应商在设计阶段确定，需求层保持供应商无关

## 需求

### 需求 1：自然语言问答交互

**目标**：作为企业内部员工，我希望用自然语言向系统提问并得到带来源的回答，以便无需手动关键字搜索就能找到内部文件中的信息。

#### 验收标准

1. When 用户在聊天界面提交一条非空的自然语言问题，the Agent 服务 shall 处理该问题并返回一段自然语言回答
2. When Agent 服务生成引用了知识库内容的回答，the Agent 服务 shall 在回答中标明其依据的来源（文档文件名及相关片段）
3. When 回答引用了某文档作为来源，the Agent 服务 shall 在回答中提供该文档原文的可点击查看链接
4. If 用户提交的问题为空或仅含空白，then the Agent 服务 shall 拒绝处理并提示用户输入有效问题
5. While Agent 服务正在处理问题，the 聊天界面 shall 向用户呈现处理中的状态指示

---

### 需求 2：会话内多轮对话

**目标**：作为用户，我希望在同一会话中能就先前的回答继续追问，以便进行连贯的多轮对话。

#### 验收标准

1. While 同一会话进行中，when 用户提交一个引用先前问答的追问，the Agent 服务 shall 在生成回答时利用该会话已有的对话历史
2. When Agent 服务向 LLM 提交请求，the Agent 服务 shall 包含当前会话的既往问答轮次作为上下文
3. The Agent 服务 shall 按会话隔离对话历史，使不同会话的上下文互不干扰
4. The 对话历史 shall 仅保存在内存中；服务重启后历史不被保留

---

### 需求 3：Agent 工具调用循环

**目标**：作为系统，我希望由 LLM 自主决定调用哪些知识库工具并循环执行，以便无需用户手动选择检索方式。

#### 验收标准

1. When Agent 服务收到用户问题，the Agent 服务 shall 将问题连同可用工具定义一并提交给 LLM
2. When LLM 返回一个或多个工具调用请求，the Agent 服务 shall 执行对应工具并将工具结果回传给 LLM
3. The Agent 服务 shall 重复"调用工具 → 回传结果"循环，直到 LLM 返回不含工具调用的最终回答
4. The Agent 服务 shall 为单次问答设置工具调用轮次上限，并在达到上限时终止循环、返回已有信息
5. If LLM 判断无需调用任何工具，then the Agent 服务 shall 直接返回 LLM 生成的回答

---

### 需求 4：知识库语义搜索工具

**目标**：作为 Agent，我希望有一个语义搜索工具，以便针对"XXX 是什么 / 如何做 XXX"类问题检索最相关的文档片段。

#### 验收标准

1. When Agent 调用语义搜索工具并提供查询文本，the 工具 shall 经由知识库检索接口返回最相关的文档片段集合
2. The 语义搜索工具返回的每个片段 shall 包含片段内容、所属文档文件名及该文档原文的查看链接
3. If 知识库检索接口返回空结果，then the 工具 shall 向 Agent 返回"无匹配内容"的明确信号而非错误

---

### 需求 5：文档列表工具

**目标**：作为 Agent，我希望有一个文档列表工具，以便回答"知识库里有哪些资料"类问题。

#### 验收标准

1. When Agent 调用文档列表工具，the 工具 shall 返回知识库中所有文档的清单，每项含文件名与处理状态
2. If 知识库中没有任何文档，then the 工具 shall 返回空清单而非错误

---

### 需求 6：文档详情工具

**目标**：作为 Agent，我希望有一个文档详情工具，以便回答关于特定文档的问题。

#### 验收标准

1. When Agent 调用文档详情工具并提供文档标识符，the 工具 shall 返回该文档的详细信息（文件名、处理状态、片段数量、原文查看链接）
2. If 指定的文档标识符不存在，then the 工具 shall 向 Agent 返回"文档不存在"的明确信号

---

### 需求 7：思考过程可观察

**目标**：作为开发者与演示者，我希望看到 Agent 的工具调用过程，以便理解其推理链路并验证 Tool Use 循环。

#### 验收标准

1. When Agent 服务执行任一工具调用，the Agent 服务 shall 将该次调用的工具名称与参数输出到运行日志
2. When 工具返回结果，the Agent 服务 shall 将结果摘要输出到运行日志

---

### 需求 8：Web 聊天界面

**目标**：作为非工程师用户，我希望有一个简单的网页聊天界面，以便无需安装任何工具即可提问。

#### 验收标准

1. When 用户访问 Web 服务的根路径，the Web 服务 shall 返回一个可输入问题并显示回答的聊天页面
2. When 用户在聊天界面提交问题，the 聊天界面 shall 将问题发送至聊天接口，并在收到回答后将其显示在对话区
3. The 聊天界面 shall 按提交顺序保留并展示当前会话内的问答记录

---

### 需求 9：MCP Server

**目标**：作为 MCP 客户端（如其他 AI 助手），我希望通过 MCP 协议使用同一套知识库工具，以便在自己的环境中复用检索能力。

#### 验收标准

1. The MCP Server shall 以 MCP 协议暴露与 Agent 相同的三个知识库工具（语义搜索、文档列表、文档详情）
2. When MCP 客户端调用其中某个工具，the MCP Server shall 执行与 Agent 内部一致的工具逻辑并返回结构化结果
3. When MCP 客户端以相同输入调用某工具，the MCP Server 返回的结果 shall 与 Agent 内部同一工具的结果一致

---

### 需求 10：错误处理与降级

**目标**：作为用户，我希望在系统出错时得到清晰反馈，而不是无响应、崩溃或被编造的答案。

#### 验收标准

1. If LLM 调用失败或超时，then the Agent 服务 shall 向用户返回一条说明暂时无法处理的错误消息，而非崩溃或返回空响应
2. If 知识库检索接口（上游 API）不可用或返回错误，then the 工具 shall 向 Agent 返回明确的错误信号，且 the Agent 服务 shall 据此向用户生成友好的失败提示
3. When 检索结果为空（知识库无相关内容），the Agent 服务 shall 明确告知用户未找到相关资料，而非编造答案

---

### 需求 11：上游知识库 API 依赖

**目标**：作为系统，我希望所有知识库访问都经由既有知识库 REST API，以便不在本服务内重复实现检索逻辑。

#### 验收标准

1. The Agent 服务 shall 通过既有知识库 REST API 完成全部知识库访问，不在本服务内实现嵌入生成或向量搜索
2. The 工具发起的请求与解析的响应 shall 与知识库 REST API 的契约（端点、字段结构、原文 `file_url`）保持一致
