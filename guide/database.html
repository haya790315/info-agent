❓ 設計上の決定

💡
なぜPinecone/Weaviateではなくpgvectorを使うのか？
DocumentメタデータはすでにPostgreSQLで管理しているため、pgvectorを使えばベクトル検索とメタデータクエリを同一トランザクション・同一接続で完結できます。独立したベクトルDBの運用コストと接続管理が不要になります。
💡
なぜDocument削除時にChunkをCASCADE DELETEするのか？
ベクトルデータは孤立すると検索精度を汚染します。Djangoのon_delete=CASCADEによってPostgreSQLレベルで原子的に削除されるため、アプリケーション層での手動クリーンアップが不要です。
✅
なぜstatus管理をDocumentモデル内に持つのか？
PDF処理（抽出→チャンク分割→ベクトル化）は時間がかかります。statusフィールドにより処理の進捗をAPIで公開でき、フロントエンドがポーリングで状態確認できます。処理失敗時はerror_messageで原因を追跡できます。

📄 Documentテーブル

PDFドキュメントのメタデータと処理ステータスを管理します。デフォルトソートは-uploaded_at（新しい順）。

カラム型制約説明
id BIGINT PK 自動採番主キー（64ビット）
filename VARCHAR(255) NOT NULL アップロードされたPDFファイル名
category VARCHAR(64) BLANK, DEFAULT='' 文書分類（resume / manual / policy / technical / report / other）
file VARCHAR BLANK, NULL PDF保存パス（media/pdfs/{category}/{filename}）
uploaded_at TIMESTAMPTZ auto_now_add アップロード日時（変更不可）
status VARCHAR(20) DEFAULT='pending' 処理状態（pending → processing → complete / failed）
error_message TEXT BLANK, DEFAULT='' 処理失敗時のエラー内容
chunk_count INT DEFAULT=0 処理完了後のチャンク総数

ステータス遷移

pending → processing → complete
pending → processing → failed

カテゴリ一覧

resume / manual / policy
technical / report / other

🔢 Chunkテーブル

PDFから抽出したテキストブロックと384次元の埋め込みベクトルを格納します。デフォルトソートはposition（文書内順序）。

カラム型制約説明
id BIGINT PK 自動採番主キー（64ビット）
document_id BIGINT FK CASCADE Document外部キー。Documentが削除されると連動して削除される
content TEXT NOT NULL テキストチャンクの内容
embedding VECTOR(384) pgvector sentence-transformersが生成した384次元のベクトル表現
position INT NOT NULL 文書内チャンクの順序（0始まり）
🔁
related_name='chunks'：DjangoORMからdocument.chunks.all()でそのドキュメントのすべてのChunkを取得できます。N+1問題はselect_related('document')で防止しています。

📋 マイグレーション履歴

0
0001_vector_extension
PostgreSQLのpgvector拡張を有効化。他のマイグレーションより先に実行する必要があります。VectorFieldを使うすべてのテーブルの前提条件です。
1
0002_initial
knowledge_base_documentテーブルを作成（filename / status / error_message / chunk_count）。knowledge_base_chunkテーブルを作成（content / embedding VECTOR(384) / position / document_id FK）。
2
0003_document_file
Documentにfileフィールドを追加（FileField, blank=True, null=True, upload_to='pdfs/'）。PDFファイル本体の保存パスを管理するために追加されました。
3
0004_document_category
Documentにcategoryフィールドを追加（CharField, 6種類の選択肢）。カテゴリ別チャンク戦略とベクトル検索フィルタリングのために追加されました。
4
0005_alter_document_file
fileフィールドのupload_toを固定文字列から動的関数_pdf_upload_pathに変更。カテゴリ別サブフォルダ（pdfs/{category}/{filename}）に自動振り分けされます。

💾 ファイルストレージ構造

アップロードされたPDFはmedia/以下にカテゴリ別に保存されます。単一ファイルの上限は10MBです。

media/
└── pdfs/
├── resume/ # 職歴書・履歴書
├── manual/ # 操作マニュアル
├── policy/ # 規程・ポリシー
├── technical/ # 技術資料
├── report/ # レポート・報告書
├── other/ # その他
└── uncategorized/ # 分類なし
⚠️
ローカルストレージの制限：現在はサーバーのローカルファイルシステムに保存されます。本番環境ではFileFieldのストレージバックエンドをAmazon S3やGoogle Cloud Storageに切り替える予定です。
