❓ 設計上の決定

💡
なぜPinecone/Weaviateなどのベクトルデータベースではなくpgvectorを使うのか？
このプロジェクトはすでにPostgreSQLでDocumentメタデータを保存しているため、pgvectorを使えばベクトル検索とメタデータクエリを同一データベースで完結でき、独立したベクトルDBサービスの追加管理が不要となり、アーキテクチャの複雑さとコストを削減できます。
💡
なぜOpenAI Embeddingsではなくparaphrasemultilingual-MiniLM-L12-v2を使うのか？
ローカルで実行できる埋め込みモデルのため、APIコストが不要で、多言語（日本語文書・中国語クエリ）にも対応しています。384次元で十分な意味類似性を捉えられ、外部API呼び出しよりも推論速度が高速です。
✅
カテゴリ別チャンクサイズの設計原則
職歴書などの短い項目は小さなChunk（400文字）を使用し、各Chunkが単一スキルや職歴を表すため検索精度が高くなります。技術資料などの長い説明は大きなChunk（1000文字）を使用し、より完全なコンテキストを保持して断片的な情報の検索を防ぎます。

🗄️ データモデル

2つのコアModel：Document（ドキュメントメタデータ＋ステータス）とChunk（テキストブロック＋384次元ベクトル）。

📄 Document
フィールド型説明
filenameCharField(255)元のPDFファイル名
categoryCharField(64)ドキュメント分類（resume/manual/policy/technical/report/other）
fileFileField元PDFの保存パス（pdfs/{category}/{filename}）
uploaded_atDateTimeFieldアップロード日時（auto_now_add）
statusCharField(20)pending / processing / complete / failed
error_messageTextField失敗時のエラーメッセージ
chunk_countIntegerField完了後のChunk総数
🔢 Chunk
フィールド型説明
documentForeignKey(Document)所属ドキュメント（CASCADE削除）
contentTextFieldテキスト断片の内容
embeddingVectorField(384)384次元コサインベクトル（pgvector）
positionIntegerFieldドキュメント内の順序（0始まり）
🗑️
Cascade Delete設計：Documentを削除すると、関連するすべてのChunk（ベクトルデータを含む）がPostgreSQLによって自動的にカスケード削除されるため、手動でベクトルデータを削除する必要がありません。

Python 
# knowledge_base/services/processor.py — チャンク分割のコアロジック
def split_into_chunks(text, chunk_size=1000, overlap=200):
    if not text:
        return []
    # stride = chunk_size - overlap → 隣接するChunkに重複部分あり
    # 重複により、Chunk境界をまたぐ情報の断裂を防ぐ
    stride = max(1, chunk_size - overlap)
    return [text[i: i + chunk_size] for i in range(0, len(text), stride)]

🔍 ベクトル検索サービス

Python 
# knowledge_base/services/searcher.py — コア検索ロジック
from pgvector.django import CosineDistance

def search(query_vector, top_k=5, category=None):
    # select_related でN+1を回避：各ChunkとDocumentデータを同時に取得
    qs = Chunk.objects.select_related('document')
    if category:
        qs = qs.filter(document__category=category)
    # CosineDistanceは昇順 = 類似度は降順（0=完全一致）
    return qs.order_by(CosineDistance('embedding', query_vector))[:top_k]

🌐 REST APIエンドポイント

メソッドパス説明
GET /api/search/ ベクトル検索。パラメータ：q（必須）、category（省略可）。JSON配列を返す（filename、content、distanceを含む）。
GET /api/documents/ ドキュメント一覧。全ドキュメントのid、filename、status、category、chunk_count、uploaded_atを返す。
GET /api/documents/<id>/ ドキュメント詳細。error_message（存在する場合）を追加で返す。
POST /（Web UI） PDFアップロード（フォーム送信）。フィールド：file（PDF）、category（省略可）。
🔗
検索レスポンスにドキュメントの元PDF URL（fileUrl）がある場合、Agentはそれを出典リンクに含め、ユーザーが元のドキュメントを直接ダウンロードできるようにします。
